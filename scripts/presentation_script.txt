SLIDE 2 — THE PROBLEM  (~2 minutes)
─────────────────────────────────────

"Right now, when a batch of IT tickets comes in, an analyst has to open
each one, read it, decide what kind of problem it is, judge how critical
it is, and then figure out which team or individual should get it.

That process typically takes 15 to 30 minutes for a batch — and that's
on a good day. The bigger problems are these:

[Point to each bullet as you say it]

First — it's slow. Every minute an analyst spends on routing is a minute
not spent on actual problem solving.

Second — it's inconsistent. Different analysts make different calls on
the same ticket. One person routes a database slowness issue to the
Network team, another sends it to the Database team. That confusion
causes rework.

Third — there's no priority intelligence. A P1 outage sitting in a
queue behind 50 low-priority requests is a real risk.

And finally — there is zero visibility. Management has no dashboard to
see which teams are overloaded or how long routing is taking.

This project solves all of these problems."


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SLIDE 3 — OUR SOLUTION  (~2 minutes)
──────────────────────────────────────

"Our solution is a multi-agent AI pipeline. Think of it as a team of
five specialised AI agents, each doing one job really well.

[Point to each box]

Agent 1 — the Classifier — reads the ticket and says: 'this is a
Database issue' or 'this is a Security problem.'

Agent 2 — the Priority Agent — reads the ticket and says: 'this is P1
Critical because the word outage appears and a system is down.'

Agent 3 — the Routing Agent — knows which team handles Database issues
and who in that team is available. It picks the right person.

Agent 4 — the Confidence Agent — is the decision maker. It combines
all the scores and asks: 'how sure are we?' If the system is more than
85% confident, it auto-routes the ticket and sends an email immediately.
If it's between 65 and 85%, it flags the ticket for a human to review.
Below 65%, it holds the ticket for manual classification.

Agent 5 — the Feedback Agent — learns from human corrections. When a
reviewer says 'this wasn't a Database issue, it was a Network issue,'
that correction is saved and used to retrain the model.

The result: more than 85% of tickets are handled without any human
touching them. Manual effort is cut by at least 50%."

SLIDE 4 — END-TO-END WORKFLOW  (~3 minutes — the KEY slide)
─────────────────────────────────────────────────────────────

[This is the most important slide. Take your time.]

"Let me walk you through the entire workflow from start to finish —
this is the heart of the system.

[Point to Step 1 — Excel File]
It starts with a simple Excel file. The user — or a scheduler — uploads
a spreadsheet of tickets. Each row is one ticket with a summary and a
description. That's all we need.

[Point to Step 2 — Web Portal]
The file is uploaded through our React web portal. It's a clean,
professional interface — think of it like a business application, not
a developer tool. You drag the file in, you press upload, done.

[Point to Step 3 — FastAPI Backend]
Behind the scenes, a FastAPI server receives the file and starts
processing each ticket through our agent pipeline.

[Point to Step 4 — AI Agents]
Each ticket goes through four agents in sequence — Classify, Prioritize,
Route, Confidence. This takes a couple of seconds per ticket.

[Point to the three outcome boxes]
Then the Confidence Agent makes a decision:

  Green box — Auto-Routed. The system is confident. It picks the
  assignee, sends an email notification, and moves on.

  Orange box — Flagged. The system isn't sure enough to act alone.
  It puts the ticket in a review queue for a human to approve.

  Red box — Held. The system doesn't know enough. A human must
  classify this one manually.

[Point to the bottom bar]
And the output? An enriched Excel file with 8 new AI columns added —
AI Category, AI Priority, the assigned team, the assignee's name,
confidence scores, routing status, and whether an email was sent.

The original Excel is untouched — we just add intelligence to it."

SLIDE 5 — TECHNOLOGY STACK  (~1.5 minutes)
────────────────────────────────────────────

"Let me quickly cover the technology — I'll keep this brief.

[Point to AI/ML box]
For the AI brain, I'm using two things. First, a machine learning model
built with scikit-learn — a standard Python library. It uses a technique
called TF-IDF to convert text into numbers, then a Logistic Regression
classifier to predict categories. Second, Ollama — a local large language
model that acts as the fallback for ambiguous tickets the ML model isn't
sure about.

[Point to Backend box]
The backend is FastAPI — a modern, fast Python web framework. The
database is SQLite locally, but it can be switched to Azure PostgreSQL
with a single environment variable change.

[Point to Frontend box]
The frontend is React with Ant Design — the same UI library used in
enterprise applications like Alibaba Cloud Console.

[Point to Cloud Action box]
The real cloud operation — one of our KRA requirements — is Gmail SMTP.
When a ticket is auto-routed, a real HTML email is delivered to the
assignee's inbox.

Everything here is open source. Zero licensing cost."

SLIDE 6 — MULTI-AGENT PIPELINE  (~1 minute — can skim if time short)
──────────────────────────────────────────────────────────────────────

"Just a quick deeper look at each agent in case you're curious.

[Point to columns]
The Classifier tries the ML model first. If the confidence is above 55%,
it uses that result. Otherwise it calls Ollama for a second opinion.

The Priority Agent has keyword rules for obvious cases — words like
'outage', 'down', 'critical', 'breach' automatically trigger P1.
Ambiguous cases go to Ollama.

The Routing Agent reads a configuration Excel file to know which team
handles which category and who's in each team. You don't need to touch
code to change routing — just update the spreadsheet.

The Confidence Agent does a weighted average — the ML score carries 50%
weight, the LLM score 30%, and priority certainty 20%. That weighted
score drives the auto-route / flag / hold decision.

The Feedback Agent saves every human correction and can trigger
retraining when enough corrections have been collected."

ANTICIPATED QUESTIONS & SUGGESTED ANSWERS
──────────────────────────────────────────

Q: What happens if the AI makes a wrong decision?
A: "The system has two safety nets. First, tickets below 85% confidence
   are never auto-routed — they go to a human reviewer. Second, when a
   reviewer corrects a decision, that correction is saved and used to
   retrain the model, so the same mistake won't happen again."

Q: What if Ollama is down or slow?
A: "The ML model handles the majority of tickets without touching Ollama.
   Ollama is only called when the ML model's confidence is below 55%.
   If Ollama is unavailable, the system falls back to the ML prediction
   alone, so it continues working."

Q: How do we control who gets emails?
A: "Everything is configured in routing_config.xlsx — team names, member
   names, and email addresses. Update the spreadsheet, restart the
   service, and the new routing takes effect immediately."

Q: Can we connect this to our existing ticketing system?
A: "Yes — that's Sprint 3 on the roadmap. The FastAPI backend exposes
   a standard REST API, so connecting to ServiceNow, Jira, or any
   system with a webhook is straightforward."

Q: What data does this system store?
A: "It stores ticket IDs, AI predictions, and human feedback corrections
   in a local SQLite database. No ticket descriptions or personal data
   are sent outside the network — Ollama runs completely locally."

Q: How accurate is it?
A: "70 to 80% baseline on 50 training examples. With 200 to 300 examples
   and a few weeks of the feedback loop running, we expect to reach and
   exceed the 85% KRA target."

Q: How long does it take to process tickets?
A: "For a batch of 50 tickets: roughly 90 seconds to 3 minutes depending
   on how many need the Ollama fallback. Tickets that the ML model handles
   are near-instant. If you train the model first, most tickets skip
   Ollama entirely."
